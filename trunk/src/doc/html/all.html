Brick Name #adapt_multi
Author Roland Niese (modified to use lock_mutex instead of lock_ulinux)
Copyright Hardy Kahl
License SOFTWARE-LICENSE, PATENT-LICENSE
Contexts
pconf !.*, ulinux.*, klinux
Purpose Allows multiple users on a single-user nest
Description 
Outputs not capable of handling multiple simultaneous operation calls can use this adapter to make it a monitor.
	Input and output calls are locked separately, so at most one input and one output call can pass this brick at the same time.
	Modification by nieserd Dez 2005:
	Brick used #lock_ulinux to implement mutual exclusion, what failed when two incoming
	simultaneous operation calls used the same mandate (#lock_ulinux is mandate-oriented).
	#lock_mutex_ulinux ignores mandates, so this won't happen anymore.
	Internally uses a #tmp_dummy_ulinux brick for lock_mutex' state. For compatibility with
	programs using the brick, no additional input is added.
	
	Note:
		Bricks should try to implement concurrency safety in a more flexible way, e.g.
		called operations could lock only the ranges of the status nest they actually access.

Brick Name #control_simple2
Author Thomas Schoebel-Theuer
Copyright University of Stuttgart
License SOFTWARE-LICENSE, PATENT-LICENSE
Contexts
Purpose (De)instantiate other bricks upon request from the control output
Description 
A control_* capable of controlling itself, i.e. running the instances
in the same address space as itself.
It has no code input, no dynamic linking, no image output.
Currently it uses malloc(), since the outermost control will always
run in a non-Athomux environment.
Future versions will use a :<mem input to store instantiated bricks.
  
Use $trans to read/write to the strategy nest, at addresses that are multiples of DEFAULT_TRANSFER (see common.h)
When a reading $trans succeeds but returns zero bytes, there is no brick instance here but there are more at higher addresses.
When a reading $trans fails, there is no brick instance here and there are no more at higher addresses.
This way you can iterate through all bricks accessible through this strategy nest.

A reading $trans returns something like this:

brick==somebrick {
	attr big=yes
	attr age=2years
	(input)=i_uninit {
		attr phio=:>o_init
		attr logio=all
		connect i_uninit=5000:outlet
	}
	input=i_init {
		attr dyn=none
		attr lock=all
		connect i_init=120000:resource
	}
	(output)=o_uninit {
		attr phio=all
		attr logio=all
		attr dyn=all
		attr addr=all
		attr lock=all
		connect o_init=3000:plug,4000:inp,8000:use
	}
	output=o_init {
		attr lock=all
	}
}

...where i_uninit, i_init, o_uninit and o_init are the connectors of the brick (arbitrary identifiers).
(input) or (output) denote an uninitialized connector.
You can:
- (dis)connect uninizialized inputs to/from both initialized and uninitialized outputs
- only initialize connected inputs
- initialize outputs, no matter whether they are connected. Initialization may fail though, if the brick has 
  uninitialized inputs vital for running the output

Purpose Access athomux nests as linux files
Description 
Nests that are connected to the image[] inputs are made accessible
    as files in the linux filesystem by use of the filesystem in
    userspace kernel module (FUSE).
    The actual filesystem part is implemented in fuse_ulinux.
    userfs_mux_ulinux redirects read/write calls to the filesystem
    to the appropriate image input.
